# CLAUDE.md

SampleOrderSystem 작업 시 Claude Code가 따르는 규칙. 요구사항/도메인 설계는 [PRD.md](PRD.md) 참고.

## 작업 진행 방식 (필독)

- 기능 요청을 받으면 바로 코드를 수정하지 않는다. 먼저 방향(요구사항 해석, 설계 옵션, 영향 범위)을 사용자와 충분히 토론한다.
- 사용자가 명시적으로 "수정해", "진행해" 등 최종 컨펌을 준 뒤에만 실제 코드 수정을 시작한다.
- 토론 없이 바로 구현하지 않는다. 방향에 대한 이견이나 모호한 부분이 있으면 구현 전에 반드시 질문한다.
- 구현 절차는 **"Plan.md/docs 갱신 → 구현"** 방식을 따른다. 엄격한 RED(테스트가 먼저 실패하는 것을 확인)-GREEN-REVIEW 사이클(테스트를 코드보다 반드시 먼저 작성하고 실패를 눈으로 확인하는 절차, 사이클마다 4개 SubAgent를 병렬로 돌리는 것)은 속도 저하 때문에 사용하지 않는다. [test-driven-development 스킬](.claude/skills/SKILL.md)도 이 프로젝트에서는 완화된 버전으로 적용된다(스킬 파일의 "프로젝트 오버라이드" 참고).

### 1. Plan.md 작성
1. 기능 요청의 목표를 정확히 정의한다(토론 포함, 사용자 컨펌 전에는 구현하지 않음).
2. [PLAN_TEMPLATE.md](PLAN_TEMPLATE.md)를 복사해 `Plan.md`를 작성한다: 목표, 접근 방식, 영향받는 파일, 확인 방법(어떻게 검증할지). 실패 테스트를 미리 나열하는 절차는 생략해도 된다.
3. 사용자 검토/승인을 받는다.

### 2. docs/ 갱신
관련 있는 `docs/<번호>-<메뉴명>.md`를 이번 변경 내용에 맞게 먼저 갱신한다(요구사항, 예외 케이스, 관련 파일 등). 코드를 구현하기 전에 문서부터 최신 상태로 맞춘다.

### 3. 구현
1. Plan.md/docs에 정리된 내용대로 코드를 구현한다.
2. 테스트는 구현과 동시에 또는 직후에 작성한다 — 실패하는 것을 먼저 확인하는 절차는 요구하지 않는다. 다만 새/변경된 기능은 최소 1개 이상의 정상 케이스 + 실패 케이스 테스트를 반드시 포함하고, 최종적으로 `python -m unittest discover tests`가 전부 통과해야 한다.
3. 구현이 끝나면 가벼운 자체 점검(아래 Harness 참고)을 한 뒤 사용자에게 결과를 보여주고 확인을 받는다.
4. 사용자 확인이 끝나면 그때 커밋한다 (`feat(...)`/`fix(...)` 등 Conventional Commits 형식).

## 구조

```
app/
  db.py          # SQLite 연결 및 스키마 초기화 (samples, orders)
  models.py      # Sample, Order 데이터클래스
  repository.py  # SampleRepository, OrderRepository: Create/Read/Update/Delete
  services.py    # OrderService: 주문 상태 전이 및 재고 검증 규칙
  cli.py         # 콘솔 메뉴 / 입출력
main.py          # 진입점
tests/           # in-memory sqlite 기반 단위 테스트
```

## 코딩 컨벤션

- 비즈니스 규칙(상태 전이, 주문 승인/출고 시점의 재고 비교·차감 같은 도메인 규칙)은 `services.py`에만 둔다. `cli.py`와 `repository.py`에는 넣지 않는다.
- 필드 자체의 구조적 유효성(생성 시점 불변식, 예: 값의 범위·양수 여부)은 `models.py`의 `__post_init__`에 둔다. 여러 엔티티에 걸친 규칙이나 시점에 따라 달라지는 규칙(예: 승인 시점 재고 비교)은 `services.py`로 간다.
- 잘못된 입력/상태는 `ValueError` 또는 `KeyError`를 던진다. 캐치는 각 메뉴 루프 자신의 최상위에서만 한다(예: 메인 메뉴는 `run()`의 루프에서, 시료 관리 하위 메뉴는 `sample_management_menu()`의 루프에서 각각 캐치). 하위 함수(`register_sample` 등)에서 미리 조건 검사 후 `print`+`return`으로 눙치지 않는다.
- 콘솔 UX: 각 메뉴 루프는 다시 그릴 때마다 화면을 지운다(`clear_screen()`, `os.system("cls")`). 목록성 데이터는 5개씩 페이지네이션(`paginate()`)하고 `N`/`B`로 이동한다. 액션 실행 후에는 결과를 보여준 채로 `pause()`(Enter 대기)한 뒤 화면을 지운다.
- 콘솔 테이블: 표는 `print_table()` 공용 헬퍼로 그린다 — 컬럼 폭에 맞춰 정렬하고, 단선 박스 문자(`┌┬┐├┼┤└┴┘─│`)로 테두리를 그리고, 헤더 행은 배경색(ANSI)으로 강조한다. Windows 콘솔(cp949) 호환성이 확인된 단선 박스 문자만 쓴다 — 이중선(`═║╔` 등)은 cp949에서 인코딩 에러가 나므로 사용하지 않는다. 새로운 이모지/특수기호를 쓰기 전에는 `ch.encode("cp949")`로 먼저 확인한다.
- 주석은 WHY가 비자명할 때만 최소한으로 작성한다. 함수/클래스 설명용 docstring은 꼭 필요한 경우가 아니면 생략한다.
- 새 기능을 추가할 때 PRD.md의 "다음 단계로 미룸" 항목에 있으면, 먼저 PRD.md를 업데이트한 뒤 구현한다.

## 테스트

- 프레임워크: `unittest` (기존 PoC와 동일하게 유지).
- 실행: `python -m unittest discover tests`
- DB 관련 테스트는 파일 DB가 아닌 in-memory sqlite(`sqlite3.connect(":memory:")`)로 격리한다.
- 새 기능은 최소 1개 이상의 실패 케이스(잘못된 상태 전이, 재고 부족 등)를 테스트에 포함한다.

```
python main.py
```

## 커밋 규칙 (Conventional Commits)

`<type>(<scope>): <description>` 형식을 사용한다.

- `feat`: 새 기능
- `fix`: 버그 수정
- `test`: 테스트 추가/수정
- `docs`: 문서(README/PRD/CLAUDE.md) 변경
- `refactor`: 동작 변화 없는 구조 개선
- `chore`: 빌드/설정 등 기타

예: `feat(order): 주문 승인/거절 상태 전이 구현`

## Harness (Claude Code 운영)

- 코드 변경 후에는 `python -m unittest discover tests`로 검증한다.
- 도메인 규칙을 바꾸는 변경은 PRD.md의 해당 표/흐름도를 함께 갱신한다.
- 커밋 전 `/code-review` 또는 동등한 리뷰로 CleanCode 위반(불필요한 추상화, 미사용 코드, 방어적 코드 과다 등) 여부를 확인한다.

### AI 선(先) 검증 원칙

AI가 만든 코드를 인간이 리뷰하기 전에, AI 스스로 충분히 테스트되어 있어야 한다. 사람은 "다 통과된 결과"를 검토하는 것이지, 통과 여부 자체를 대신 확인해주는 역할이 아니다.

- **기본**: 동작 검증(Correctness) — `python -m unittest discover tests`로 새/변경된 기능이 요구사항대로 동작하는지 증명한다.
- **필요 시**: 사람이 최종 확인해야 할 항목(콘솔 인터랙션 등 자동 테스트로 잡기 번거로운 부분)을 AI가 먼저 선행 검증해서, 리뷰 시점에는 "확인만 하면 되는" 상태로 만든다.
- **필요 시**: `/tmp` 등 격리된 경로에서 공격적인 Safety Test(비정상 입력, 경계값, 악의적 입력 등)를 수행한다.

### 가벼운 자체 점검 (사람 리뷰 전, 매 사이클)

이전에는 이 단계를 4개 SubAgent를 매번 병렬로 호출해 수행했으나, 사이클당 소요 시간이 길어 속도 저하가 컸다. 지금은 **AI(주 스레드)가 직접, 한 번에** 아래 4가지 관점을 훑고 문제를 바로 고친 뒤 사람에게 결과를 보여준다(`.claude/agents/`의 4개 정의 파일은 각 관점의 체크리스트로 계속 참고하되, 별도 SubAgent 호출은 강제하지 않는다):

1. **문서 정합성**: PRD.md/README.md/docs/*.md가 실제 구현과 어긋나지 않는가.
2. **범위 준수**: Plan.md에 없는 변경(리팩터링, 기능 추가 등)이 섞이지 않았는가.
3. **테스트**: 전체 테스트가 통과하는가, 새 기능에 실패 케이스가 있는가.
4. **컨벤션 준수**: 코딩 컨벤션(비즈니스 규칙 위치, 예외 처리, 커밋 규칙 등)을 지켰는가.

이 점검에서 발견한 문제는 사람에게 묻지 않고 바로 고쳐도 되는 수준(오탈자, 문서 누락 등)이면 즉시 반영하고, 설계 판단이 필요한 사안이면 사람에게 확인을 구한 뒤 반영한다. 점검이 끝나면 결과를 요약해 사람에게 보여주고, 승인이 나야 커밋을 진행한다. 4개 SubAgent를 실제로 병렬 호출하는 것은 사용자가 명시적으로 요청할 때만 한다.
