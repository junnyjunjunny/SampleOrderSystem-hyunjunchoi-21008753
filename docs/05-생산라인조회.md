# 5. 생산 라인 조회

## 개요

`[5] 생산 라인 조회`는 하위 메뉴 없이 `[0] 뒤로`만 있는 단일 화면이다. 진입 중에는 약 0.3초 간격으로 화면을 자동 갱신하며(`clear_screen()` + 다시 그리기), 매 갱신마다 생산 라인을 한 틱(tick) 진행시킨다 — 그래서 진행 게이지/완료율이 실시간으로 올라가는 것처럼 보인다.
생산 라인은 1개(단일 라인)이며, 동시에 하나의 주문만 "현재 처리 중"이 될 수 있다. 나머지 `PRODUCTION` 상태 주문은 "대기 중"으로 순서대로 표시된다.
`[0]`은 Enter 없이 키를 누르는 즉시 반응한다(Windows 콘솔 `msvcrt.kbhit()`/`getch()` 기반 논블로킹 폴링).

## 도메인 모델 변경

`Order`에 필드 3개 추가(모두 nullable, 기본값 `None`):
- `production_quantity: int` — 실 생산량. `PRODUCTION`으로 전이되는 시점(`decide()`)에 `ceil(부족분 / 수율)`로 계산해 저장. 부족분 = `order.quantity - 결정 시점의 sample.stock`.
- `production_started_at: str`(ISO, UTC) — 이 주문이 "현재 처리 중"이 된 시각. 결정 시점이 아니라, 실제로 생산 라인 차례가 되었을 때(`advance_production_line()`) 설정된다.
- `production_queue_seq: int` — 생산 대기열 진입 순번. `decide()`가 `PRODUCTION`으로 전이시키는 시점에 `order_repo.next_production_queue_seq()`(현재 최댓값+1)로 부여한다. 대기열 FIFO 정렬에 쓰인다.

기존 DB(`samples.db`)에는 이 컬럼들이 없으므로 `app/db.py`에서 `PRAGMA table_info`로 존재 여부를 확인해 없으면 `ALTER TABLE`로 추가한다(기존 데이터 보존).

> **왜 타임스탬프가 아니라 정수 순번인가**: 처음에는 `production_queued_at`(ISO 타임스탬프)으로 FIFO를 구현했으나, 같은 사이클 안에서 연달아 `decide()`가 호출되면(예: 승인 화면에서 여러 건을 빠르게 승인) 이 환경의 시스템 시계 해상도가 낮아 서로 다른 주문이 완전히 동일한 타임스탬프를 받는 경우가 실제로 발생했다(테스트로 재현·확인함). 정수 순번은 그런 동시성 문제 없이 항상 유일하고 엄격하게 증가한다.

## 상태 전이/진행 로직 (OrderService)

- `decide()`가 재고 부족으로 `PRODUCTION`을 선택하면, `production_quantity`를 계산해 저장하고, `advance_production_line()`을 호출해 라인이 비어 있으면 즉시 시작되게 한다.
- `advance_production_line()`:
  1. `PRODUCTION` 상태 주문을 전부 조회한다.
  2. `production_started_at`이 있는 주문을(있다면 하나만 있어야 함) "현재 처리 중"으로 본다.
  3. 현재 처리 중인 주문이 있으면: 경과 시간(`now - production_started_at`)과 총 소요 시간(`production_quantity * sample.avg_production_time`분)을 비교한다. 경과 ≥ 총 소요면 `complete_production()`으로 완료 처리(재고를 `production_quantity`만큼 보충)하고 라인을 비운다.
  4. 라인이 비어 있으면(처음부터 없었거나 방금 비웠으면): 대기 중인 주문(`production_started_at`이 없는 `PRODUCTION` 주문) 중 `production_queue_seq`가 가장 작은 것(= 가장 먼저 승인되어 대기열에 들어온 것, FIFO)을 골라 `production_started_at`을 지금 시각으로 설정해 "현재 처리 중"으로 만든다. `production_queue_seq`가 없는 레거시 주문은 정렬에서 뒤로 밀리며, 그런 주문끼리는 `created_at`으로 순서를 매긴다. 이때 `production_quantity`가 없는 레거시 주문(이 기능 이전에 이미 `PRODUCTION`이었던 주문)이면 현재 재고 기준으로 계산해 채워 넣는다(하위 호환).
     - **FIFO 기준은 주문 접수 시각(`created_at`)이 아니라 승인(=`PRODUCTION` 진입) 시각이다.** 먼저 접수됐지만 승인이 늦어진 주문은, 나중에 접수됐지만 먼저 승인된 주문보다 대기열에서 뒤로 밀린다.
  5. 이 메서드는 몇 번을 호출해도 안전하다(멱등) — `[5]` 화면 진입마다 호출한다.
- `complete_production()`은 재고 보충량을 기존 `order.quantity`가 아니라 `order.production_quantity`(수율 반영한 실 생산량)로 수정한다 — 기존 로직은 수율을 무시하고 과잉 생산하는 버그였다.

## 화면 구성

```
생산라인 1 | 현재 상태: RUN

현재 처리 중
  주문번호: O-20260715-008
  시료: SiC 파워기판-6인치 (평균 생산시간 1.30 min/ea)
  주문량: 100   재고: 50   부족: 50   실 생산량: 63
  진행: [##########----------] 50%   완료 예정: 2026-07-15 15:42

대기 중인 주문
┌──────┬────────────────┬ ... ┐
│ 순서 │ 주문번호        │ ... │
├──────┼────────────────┼ ... ┤
│ ...  │ ...            │ ... │
└──────┴────────────────┴ ... ┘

[0] 뒤로 >
```

- "현재 상태": 현재 처리 중인 주문이 있으면 `RUN`, 없으면 `STOP`.
- "현재 처리 중" 블록: 주문번호 / 시료명(+평균 생산시간 min/ea) / 주문량 / 재고(현재 값, live) / 부족(`production_quantity * yield_rate`로 역산, 반올림) / 실 생산량(`production_quantity`) / 진행 게이지(`#`/`-` 20칸, ASCII만 사용 — cp949에서 `█`/`░` 등 블록 문자는 인코딩 에러 확인됨) / 완료율(%) / 완료 예정 시각(로컬 시간으로 변환해 표시).
- "대기 중인 주문" 표: 컬럼 `순서 / 주문번호 / 시료 / 주문량 / 부족분 / 실생산량 / 예상 완료`. 예상 완료는 단일 라인이 순차 처리한다고 가정하고(현재 처리 중 주문의 완료 예정 시각 → 그다음 대기 주문 순서대로 누적) 계산한다.
- 진행률/완료 예정 시각은 모두 시료의 `avg_production_time`(분/개) 기준으로 계산한다: 총 소요 시간(분) = `production_quantity * avg_production_time`.

## 상세 요구사항

### 계산 규칙
- 부족분(결정 시점) = `주문 수량 - 그 시점의 재고`
- 실 생산량 = `ceil(부족분 / 수율)` (예: 수율 0.9, 부족분 90 → 실 생산량 100). 올림 처리라 화면에 역산해 보여주는 부족분(`실생산량 * 수율`)이 원래 부족분과 반올림 오차만큼 다를 수 있음(허용).
- 총 소요 시간(분) = `실 생산량 * 시료.avg_production_time`
- 진행률(%) = `min(100, 경과분 / 총소요분 * 100)`
- 완료 예정 시각 = `production_started_at + 총소요분`

### 예외 케이스
- 시료가 삭제된 경우 시료명에 `(삭제된 시료)` 표시(기존 화면들과 동일 패턴).
- 대기 주문이 없고 현재 처리 중도 없으면 "현재 상태: STOP", 두 섹션 모두 "없습니다" 메시지.

## 데이터/상태

`Order`에 `production_quantity`, `production_started_at` 추가. 상태 흐름 자체(`RECEIVED→{CONFIRMED,PRODUCTION,REJECTED}`, `PRODUCTION→CONFIRMED`, `CONFIRMED→RELEASE`)는 변경 없음 — `PRODUCTION` 내부에서 "현재 처리 중 vs 대기 중" 구분이 새로 생긴 것뿐.
