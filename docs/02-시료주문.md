# 2. 시료 주문

## 개요

`[2] 시료 주문` 메뉴는 새 주문을 접수하는 기능만 담당한다. 접수된 주문은 `RESERVED` 상태로 저장되고 끝난다.
승인/거절 판단은 여기서 하지 않으며, 전적으로 3번(주문 승인/거절) 메뉴에서 이후에 진행한다.

## 현재 구현 상태

- 주문 접수: 구현됨 (`order_reception_menu`, `receive_order`)
- `order_id`는 사용자가 입력하지 않고 시스템이 자동 생성

## 하위 메뉴 구조

`[2] 시료 주문` 진입 시 목록 없이 바로 메뉴만 보여준다:

```
[1] 주문 접수  [0] 뒤로 > 
```

- `1`: 시료ID/고객명/수량을 입력받아 주문을 접수(주문번호 자동 생성, 상태 `RESERVED`)하고 "주문 접수가 완료되었습니다."를 보여준 뒤 끝난다.
- `0`: 메인 메뉴로 복귀

액션 실행 후에는 결과를 보여준 채로 "계속하려면 Enter"로 대기했다가, 다시 `[1] 주문 접수  [0] 뒤로` 화면으로 돌아간다.

## 주문번호 생성 규칙

- 형식: `O-YYYYMMDD-NNN` (예: `O-20260715-001`)
- `YYYYMMDD`는 접수 시각(로컬 날짜) 기준, `NNN`은 그 날짜 안에서 001부터 증가하는 3자리 일련번호.
- 하루 안에서 유일성 보장. DB 조회 기반이라 앱을 재시작해도 이어진다.

## 상세 요구사항

### 입력값 (접수)
| 필드 | 타입 | 제약 |
|---|---|---|
| sample_id | str | 필수, 존재하는 시료여야 함 |
| customer_name | str | 필수 |
| quantity | int | 1 이상 |

### 정상 흐름
1. 시료ID/고객명/수량을 입력받는다.
2. `OrderService.receive_order`로 주문을 접수한다(상태 `RESERVED`, 주문번호 자동 생성).
3. "주문 접수가 완료되었습니다. 주문번호: ..."를 보여주고 끝난다.

### 예외 케이스
- 존재하지 않는 `sample_id`로 접수 시 → `KeyError`
- `quantity`가 0 이하 → `ValueError`

## 데이터/상태

`Order(order_id, sample_id, customer_name, quantity, status, created_at)`. 상태 흐름 전체는 [PRD.md](../PRD.md) 참고.
`order_id`는 이 메뉴가 자동 생성하며 더 이상 사용자가 직접 입력하지 않는다.
