# 3. 주문 승인/거절

## 개요

2번(시료 주문) 메뉴에서 접수되어 `RESERVED` 상태로 저장된 주문을 승인하거나 거절하는 기능.
승인을 선택하면 그 시점의 재고와 주문 수량을 비교해 자동으로 `CONFIRMED`(재고 충분) 또는 `PRODUCING`(재고 부족)으로 갈린다.
거절을 선택하면 `REJECTED`로 전이한다. 결정이 끝난 주문은 이 메뉴의 목록에서 더 이상 보이지 않는다.

메인 메뉴의 기존 `[3] 주문 승인`/`[4] 주문 거절` 두 항목을 `[3] 주문 승인/거절` 하나로 합친다.

## 현재 구현 상태

- 목록 조회(RESERVED, 페이지네이션): 구현됨 (`order_approval_menu`)
- 승인/거절 결정(재고에 따른 자동 분기): 구현됨 (`OrderService.decide`)

## 하위 메뉴 구조

`[3] 주문 승인/거절` 진입 시:
1. 화면을 지운다(`cls`).
2. `RESERVED` 상태 주문 목록을 5개씩 페이지네이션해서 보여준다(시료 관리와 동일한 UX).
   컬럼: **번호(현재 페이지 내 1부터) / 주문번호 / 고객 / 시료 / 수량 / 상태**
3. 통합 프롬프트: `[N] 다음  [B] 이전  번호 입력(승인/거절)  [0] 뒤로 > `
   - 목록에 표시된 번호를 입력하면 해당 주문에 대해:
     - 시료명 / 현재 재고 / 주문 수량 / 부족분(`현재 재고 - 주문 수량`)을 보여준다. 부족분이 음수이면 빨간 글씨(ANSI 이스케이프)로 표시한다.
     - `[Y] 승인  [N] 주문 거절  [0] 뒤로` 선택을 받는다.
       - `Y`: `OrderService.decide(order_id, approve=True)` 호출 — 재고 충분하면 `CONFIRMED`로 전이하며 그 즉시 `stock -= quantity`, 부족하면 `PRODUCING`으로 전이(재고는 나중에 생산 완료 시점에 반영, `docs/05` 참고).
       - `N`: `OrderService.decide(order_id, approve=False)` 호출 — `REJECTED`로 전이.
       - `0`: 아무 것도 하지 않고 목록으로 돌아간다(주문 상태 변화 없음, "계속하려면 Enter" 대기 없이 바로 복귀).
   - `0`(목록 화면에서): 메인 메뉴로 복귀

결정(`Y`/`N`)을 내린 경우에만 결과를 보여준 채로 "계속하려면 Enter"로 대기했다가 목록 화면(1페이지)으로 돌아간다. 방금 결정한 주문은 더 이상 `RESERVED`가 아니므로 목록에서 사라진다.

## 상세 요구사항

### 입력값
| 필드 | 타입 | 제약 |
|---|---|---|
| 번호 | int | 현재 페이지에 표시된 항목 범위(1~페이지 항목 수) 안이어야 함 |

### 예외 케이스
- 페이지 범위를 벗어난 번호 입력 → `ValueError`
- `Y`/`N`/`0` 외의 값 입력 → `ValueError`
- (구조적으로 발생 안 함) 이미 결정된 주문을 다시 결정하려는 시도 — 목록에 아예 나타나지 않으므로 선택 자체가 불가능

## 데이터/상태

상태 전이 규칙: `RESERVED → {CONFIRMED, PRODUCING, REJECTED}`. 자세한 흐름도는 [PRD.md](../PRD.md) 참고.
`OrderService.decide`가 재고 비교(`sample.stock >= order.quantity`)로 `CONFIRMED`/`PRODUCING`을 자동 결정한다.
**재고 차감은 6번(출고 처리)이 아니라 여기서 일어난다** — `CONFIRMED`로 전이하는 즉시 `stock -= quantity`.

## 더미 데이터 시딩

페이지네이션/승인 자동 분기(CONFIRMED/PRODUCING)를 넉넉한 데이터로 검증하기 위해 아래 명령으로 `RESERVED` 주문 100건을 추가할 수 있다.

```
py scripts/seed_bulk_orders.py
```

23개 시료를 순환하며 수량을 다양화해 재고 충분(즉시 승인 가능)/부족(생산 필요) 케이스가 섞이도록 만든다. `OrderService.receive_order`를 통해 생성하므로 주문번호도 정상 자동 생성된다. 이벤트성 데이터라 재실행하면 100건이 추가로 더 쌓인다(멱등 아님, `seed_orders.py`와 동일한 특성).
